Tomcat IO

发表于 2015-04-04   |   分类于 Tomcat   |  

Tomcat的三种IO策略:BIO,NIO,APR

  • BIO是Tomcat默认开启的IO模式，性能较低，没有经过优化处理

  • NIO，利用Java的非阻塞技术。开启方法：将tomcat子目录下面的server.xml修改一下，将中的protocol改为org.apache.coyote.http11.Http11NioProtocol

  • APR，开启方法:将protocol改为org.apache.coyote.http11.Http11AprProtocol，还需要下载APR模式需要的库，因为这种模式是基于JNI的，即JAVA调用本地库的方式来进行IO

设计模式学习笔记

发表于 2015-03-25   |   分类于 Design Pattern   |  

  • OO基础:抽象、封装、继承、多态

  • 设计模式：模式被认为是历经验证的OO设计经验，模式不是代码，而是针对设计问题的通用解决方案

  • 在设计模式中，实现一个接口泛指实现某个超类型，可以是类或接口，的某个方法

  • 设计原则一：找出应用中可能需要的变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起(封装变化)

  • 设计原则二：针对接口编程，而不是针对实现编程
  • 设计原则三：多用组合，少用继承
  • 设计原则四：为了交互对象之间的松耦合而努力
  • 设计原则五：开放闭合原则(Open Closed Principle,OCP)类应该对扩展开放，对修改关闭
  • 设计原则六：要依赖抽象，不要依赖具体类

  • 设计原则七：最少知识原则(即得墨忒耳法则Law of Demeter，只和你的密友谈话

    • 在对象的方法内，应只调用属于以下范围的方法
      • 该对象本身的方法
      • 被当作方法的参数而传递进来的对象的方法
      • 此方法所创建或实例化的任何对象的方法
      • 对象的任何组件的方法
      • 组件是指实例变量所引用的任何对象
    • 不应调用以下范围的方法
      • 如果某对象是调用其他方法的返回结果，不应调用该对象的方法

  • 设计模式一：策略模式；定义了算法族，分别封装起来，让它们可以互相替换此模式让算法的变化独立于使用算法的客户

  • 设计模式二：观察者模式；定义了对象之间的一对多依赖，这样一来，当一个对象状态改变时，它的所有依赖者都会收到通知并自动更新
    • 定义了对象之间一对多的关系
    • 主题(可观察者)用一个共同的接口来更新观察者
    • 观察者和可观察者之间用松耦合(loosecoupling)的方式结合
    • 使用此模式可以从观察者处推或拉数据(推的方式被认为更正确)
    • 有多个观察者时,不应该依赖于观察者被通知的次序
    • Java内置了观察者模式(java.util.Observable),它是一个类而不是一个接口,这就造成了一些实现上的问题
    • swing、许多GUI框架、JavaBeans、RMI大量使用观察者
  • 设计模式三：装饰者模式；动态地将责任附加到对象上。若要扩展功能，装饰者提供了比继承更有弹性的替代方案
    • 不改变接口，但加入责任
  • 设计模式四：工厂方法模式；定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法让类把实例化推迟到子类。
  • 设计模式五：抽象工厂模式；
  • 设计模式六：单例模式；确保一个类只有一个实例，并提供一个全局访问点
  • 设计模式七：命令模式；将“请求”封装成对象，以便使用不同的请求、队列或者日志来参数化其他对象。命令模式也支持可撤销的操作。
  • 设计模式八：适配器模式；将一个类的接口，转换成客户希望的另一个接口。适配器让原本不兼容的类可以合作无间。
    • 包括对象适配器和类适配器，对象适配器使用组合，类适配器使用多重继承，不适用于Java
    • 将一个接口转成另一个接口
  • 设计模式九：外观模式；提供了一个统一的接口，用来访问子系统中的一群接口。外观定义了一个高层接口，让子系统更容易使用。
    • 让接口更简单
    • 提供简化的接口的同时，依然将系统的完整功能暴露出来，以供需要的人使用

Git提交代码到Github

发表于 2015-02-27   |   分类于 Git   |  
  1. 初始化代码仓库

    git init

  2. 添加要提交的文件到仓库的缓存区

    git add filename

  3. 将添加的文件从缓存区提交到HEAD（最近一次提交的结果）

    git commit -m “代码提交注释”

  4. 将本地代码仓库中代码push到github上

    • 首先需要用在github上的邮箱与用户名生成一个.ssh(id_rsa.pub)

      git config –global user.name “your name” (Sets the default name for git to use when you commit)
      git config –global user.email “your email” (Sets the default email for git to use when you commit)

    • 然后在git中创建主干线(ssh方式)

      • ssh方式:git remote add origin git@github.com:username/path.git
      • https方式:git remote add origin https://github.com/username/path.git(这样创建会在每次提交时都输入用户名和密码)
    • push到github中的同名项目中

      git push origin master

  5. 将github上的代码更新到本地

    git pull origin master

  6. 从github上复制项目到本地

    git clone https://github.com/username/path.git master

编码相关

发表于 2015-02-08   |   分类于 Basic   |  
  • ASCII
    • 1个字节表示1个字符
  • UNICODE
    • 2个字节表示1个字符
    • 计算机内存中统一使用UNICODE编码,当需要保存到硬盘或传输的时候转换为其他编码
  • UTF-8
    • 可变长编码
    • ASCII编码可被看成是UTF8编码的一部分
    • 把一个Unicode字符根据不同的数字大小编码成1－6个字节
    • 英文字母被编码成1个字节
    • 常用汉字通常是3个字节
    • 生僻字符被编码成4-6个字节

触发JVM full gc的情况

发表于 2015-01-22   |   分类于 Java   |  

  • 旧生代空间不足

    • 旧生代空间只有在新生代对象转入及创建为大对象、大数组时才会出现不足的现象，当执行full gc后空间仍然不足，则抛出如下错误:java.lang.OutOfMemoryError:java heap space.为避免以上两种状况引起的full gc，调优时应尽量做到让对象在minor gc阶段被回收、让对象在新生代多存活一段时间及不要创建过大的对象及数组

  • 永生代空间满

    • 永生代(permanent generation)中存放的为一些class的信息等，当系统中要加载的类、反射的类和调用的方法较多时，permgen可能会被占满，在未配置为cms gc的情况下会执行full gc.如果full gc仍然回收不了，那么JVM会抛出以下错误信息:java.lang.OutOfMemoryError:PermGen space。为避免Perm Gen占满造成Full Gc现象，可采用的方法为增大Perm Gen空间或转为使用CMS GC

  • CMS GC时出现promotion failed和concurrent mode failure

  • 统计得到的minor GC晋升到旧生代的平均大小大于旧生代的剩余空间

mysql主从同步

发表于 2015-01-15   |   分类于 Mysql   |  
  • 实现机制
    • mysql主从同步是在主从复制基础上实现的，通过设置在Master Mysql上的binlog(使其处于打开状态)，slave mysql上通过一个I/O线程从mysql上读取binlog，然后传输到slave mysql的中继日志中，然后slave mysql的sql线程从中继日志中读取中继日志，然后应用到slave mysql的数据库中。这样就实现了主从同步。
  • 作用
    • 可以作为一种备份机制，相当于热备份
    • 可以用来作读写分离，均衡数据库负载
  • 步骤
    • 准备
      • 主从数据库版本一致
      • 主从数据库数据一致
    • 主数据库修改
      • 修改mysql配置
        • log-bin=mysql-bin # 日志文件名
        • server-id=1 # 主数据库端ID号
      • 重启mysql，创建用于同步的账户
        • grant replication slave on . to ‘xxx’@’%’ identified by ‘xxx’;
        • flsuh privileges;
      • 查询master状态
        • show master status;
    • 从数据库修改
      • 修改mysql配置
        • server-id=2 # 从数据库端ID号
      • 执行同步命令
        • change master to master_host=’master_ip’,master_user=’master_user’,master_password=’master_pwd’ master_log_file=’a’,master_log_pos=b; # 其中a,b为show master status看到的File和Position值
        • start slave;
      • 检查slave状态
        • show slave status\G; # slave_io_running及slave_sql_running必须运行正常，否则同步失败
  • 其他相关参数
    • master
      • 不同步的数据库
        * binlog-ignore-db=库名  # 可分多条设置多个
      • 只同步哪些数据库
        * binlog-do-db=库名   # 开启后只同步这些数据库
      • 日志保留时间
        * expire_logs_days=10
      • 控制binlog写入频率
        * sync_binlog=5  # 每执行多少次事务写入一次，这个参数性能消耗很大，但可减小mysql崩溃损失
      • 日志格式，寻访mixed
        * binlog_format=mixed 
* statement  # 保存sql语句
* row  # 保存影响记录数据
* mixed  # 前面两种的混合
    • slave
      • 停止同步
        * stop slave;
      • 连接断开时重新连接超时时间
        * change master to master_connect_retry=50;

死锁问题

发表于 2015-01-15   |   分类于 Java   |  

####导致死锁的程序示例

原因分析:t1在获取obj1锁后休眠1秒钟，而在这段时间内，t2和t3已经先后启动并分别获取了obj2和obj3的锁，这样t1休眠结束后想要获取obj2的锁就必须等待t2获取obj3的锁，而t2要想获取obj3的锁则要等待t3获取obj1的锁，造成了循环等待

####死锁条件

  • 互斥条件:资源不能共享，只能由一个进程使用

  • 请求保持条件:进程已获得了一些资源，但因请求其他资源被阻塞时，对已获得的资源保持不放

  • 不可抢占条件:有些系统资源是不可抢占的，当某个进程已获得这种资源后，系统不能强行收回，只能由进程使用完后自己释放

  • 循环等待条件:若干进程形成环形链，每个都占用对方申请的下一个资源

####java多线程死锁原因

  • 多个线程涉及到多个锁，这些锁存在着交叉，可能会导致一个锁依赖的闭环

  • 默认的锁申请操作是阻塞的

####避免死锁

  • 避免嵌套封锁

  • 只对有请求的进行封锁

  • 避免无限期等待

####竞态条件

  • 产生

    • ﻿当两个线程竞争同一资源时，如果对资源的访问顺序敏感，就称存在竞态条件，导致竞态条件发生的代码区称为临界区

  • 避免

    • 在临界区中使用适当的同步可以避免竞态条件
