ReentrantLock与Synchronized区别

发表于 2015-06-12   |   分类于 Java   |  

  1. 用法区别

    • synchronized:在需要同步的对象中加入此控制，synchronized可以加在方法上，也可以加在特定代码块中，括号中表示需要锁的对象;

      • 无法中断一个等候锁的线程

      • 无法通过投票获得锁

      • 释放锁的操作只能在获得锁所在的代码块中进行，无法在别的代码块中释放锁

    • lock:需要显式指定起始位置和结束位置，多线程中必须要使用一个ReentrantLock类作为对象才能保证锁的生效，且在加锁和解锁处需要通过lock()和unlock()显式指出，一般在finally块中写unlock以防止死锁;ReentrantLock获取锁的几种方式，

      • lock, 如果获取了锁立即返回，否则处于休眠状态，直到获取锁

      • tryLock，如果获取了锁立即返回true，否则返回false

      • tryLock(long timeout, TimeUnit unit)，如果获取了锁立即返回true，否则等待给定时间，等待中获取锁返回true，等待结束未获取锁返回false

      • lockInterruptibly，如果获取了锁立即返回，否则处于休眠直到获取锁或被别的线程中断

    • 结论:synchronized是在JVM层面上实现的，如果在代码执行时出现异常，JVM会自动释放锁；lock是通过代码实现的，要保证锁一定会被释放，就必须将unlock放到finally块中

  2. 性能区别

    • synchronized:采用的是CPU悲观锁机制，即线程获得的是独占锁。独占锁意味着其他线程只能依靠阻塞来等待当前线程释放锁。而在CPU转换线程阻塞时会引起线程上下文切换，当有很多线程竞争锁的时候，会引起CPU频繁的上下文切换导致效率很低

    • lock:用的是乐观锁方式。乐观锁即假设没有冲突而去完成某项操作，如果因为冲突导致失败就重试，直到成功为止。乐观锁实现的机制CAS操作(Compare and Swap)。

    • 结论:在资源竞争不是很激烈的情况下，synchronized性能要优于lock，在资源竞争很激烈的情况下，synchronized的性能迅速下降，而reentrantlock性能将维持常态

  3. 用途区别

两者在一般情况下的用途没有差别，但是在复杂的同步应用中，应考虑使用ReentrantLock，因为lock提供了多样化的同步，

  • 定时锁等候-在等候时间内未获取锁，线程会自己中断

  • 可中断锁等候-线程等候可以自己中断，也可以由别的线程中断

  • 锁投票

public void quickSort(int[] arr, int l, int r) {

    if (l < r) {

        int i = l, j = r, x = arr[l];

        while (i < j) {

            while (i < j && arr[j] >= x) {

                j--;

            }

            if (i < j) {

                arr[i++] = arr[j];

            }

            while (i < j && arr[i] < x) {

                i++;

            }

            if (i < j) {

                arr[j--] = arr[i];

            }

        }

        arr[i] = x;

        quickSort(arr, l, i - 1);

        quickSort(arr, i + 1, r);

    }

}

匿名内部类

发表于 2015-05-07   |   分类于 Java   |  

  • 注意事项

    • 使用匿名内部类时，必须继承一个类或实现一个接口，但不可同时继承和实现，同时也只能实现一个接口

    • 匿名内部类中不能定义构造函数(可以使用构造代码块来初始化内部类成员)

    • 匿名内部类中不能存在任何静态成员变量和静态方法

    • 匿名内部类为局部内部类，局部内部类的所有限制对匿名内部类同样生效

    • 匿名内部类不能为抽象类，必须实现继承类或实现接口的所有方法

    • 给匿名内部类传递参数的时候，如果该参数在内部类使用，那么该参数必须为final.即当所在方法的参数需要被内部类使用时该参数必须为final

      • 原因:内部类编译成功后，会产生一个class文件，该class文件与外部类并不是同一class文件，仅仅只保留对外部类的引用，内部类并不是直接调用方法传递的参数，而是利用自身的构造器对传入的参数进行备份，自己内部方法调用的实际上是自己的属性而不是外部方法传进来的参数.简单理解:拷贝引用，为了避免引用值发生改变，例如被外部类的方法修改，而导致内部类得到的值不一致，于是让final来让该引用不可改变.所以如果定义了一个匿名内部类，并且希望它使用一个其外部定义的参数，那么编译器会要求该参数引用是不可变的

Java线程池

发表于 2015-04-19   |   分类于 Java   |  

####线程池作用

  • 降低资源消耗

    • 通过重复利用已创建的线程降低线程创建和销毁造成的消耗

  • 提高响应速度

    • 当任务到达时，任务可以不需要等到线程创建就能立即执行

  • 提高线程的可管理性

    • 无限制创建线程不仅会消耗资源还会降低系统的稳定性，使用线程池可以进行统一的分配、调优和监控

####线程池处理流程

处理流程如下图：

  1. 首先判断基本线程池是否已满，没满，创建一个工作线程(线程池创建时会将线程封装成工作线程Worker,worker在执行完当前任务时还会无限循环获取工作队列里的任务来执行)来执行任务；满了，则进入下个流程

  2. 其次判断工作队列是否已满，没满则将新提交的任务存储在工作队列里；满了，则进入下个流程

  3. 最后判断整个线程池是否已满，没满则创建一个新的工作线程来执行任务；满了，则交给饱和策略来处理这个任务

Hibernate关联

发表于 2015-04-15   |   分类于 Hibernate   |  

Hibernate鼓励使用双向一对多关联，不使用单向一对多关联。单向一对多关联映射是在one端维护的，必须先保存many端后才能保存one端。保存many端时不知道one端是否存在相应的数据，所以只能能维护的关系字段设置为null，如果非空则无法保存。因为是one端维护关系，所以在保存one端时，会发现多余的update语句维护many端的外键关系
Hibernate鼓励使用多对一关系取代一对一关系
