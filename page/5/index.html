mysql主从同步

发表于 2015-01-15   |   分类于 Mysql   |  
  • 实现机制
    • mysql主从同步是在主从复制基础上实现的，通过设置在Master Mysql上的binlog(使其处于打开状态)，slave mysql上通过一个I/O线程从mysql上读取binlog，然后传输到slave mysql的中继日志中，然后slave mysql的sql线程从中继日志中读取中继日志，然后应用到slave mysql的数据库中。这样就实现了主从同步。
  • 作用
    • 可以作为一种备份机制，相当于热备份
    • 可以用来作读写分离，均衡数据库负载
  • 步骤
    • 准备
      • 主从数据库版本一致
      • 主从数据库数据一致
    • 主数据库修改
      • 修改mysql配置
        • log-bin=mysql-bin # 日志文件名
        • server-id=1 # 主数据库端ID号
      • 重启mysql，创建用于同步的账户
        • grant replication slave on . to ‘xxx’@’%’ identified by ‘xxx’;
        • flsuh privileges;
      • 查询master状态
        • show master status;
    • 从数据库修改
      • 修改mysql配置
        • server-id=2 # 从数据库端ID号
      • 执行同步命令
        • change master to master_host=’master_ip’,master_user=’master_user’,master_password=’master_pwd’ master_log_file=’a’,master_log_pos=b; # 其中a,b为show master status看到的File和Position值
        • start slave;
      • 检查slave状态
        • show slave status\G; # slave_io_running及slave_sql_running必须运行正常，否则同步失败
  • 其他相关参数
    • master
      • 不同步的数据库
        * binlog-ignore-db=库名  # 可分多条设置多个
      • 只同步哪些数据库
        * binlog-do-db=库名   # 开启后只同步这些数据库
      • 日志保留时间
        * expire_logs_days=10
      • 控制binlog写入频率
        * sync_binlog=5  # 每执行多少次事务写入一次，这个参数性能消耗很大，但可减小mysql崩溃损失
      • 日志格式，寻访mixed
        * binlog_format=mixed 
* statement  # 保存sql语句
* row  # 保存影响记录数据
* mixed  # 前面两种的混合
    • slave
      • 停止同步
        * stop slave;
      • 连接断开时重新连接超时时间
        * change master to master_connect_retry=50;

死锁问题

发表于 2015-01-15   |   分类于 Java   |  

####导致死锁的程序示例

原因分析:t1在获取obj1锁后休眠1秒钟，而在这段时间内，t2和t3已经先后启动并分别获取了obj2和obj3的锁，这样t1休眠结束后想要获取obj2的锁就必须等待t2获取obj3的锁，而t2要想获取obj3的锁则要等待t3获取obj1的锁，造成了循环等待

####死锁条件

  • 互斥条件:资源不能共享，只能由一个进程使用

  • 请求保持条件:进程已获得了一些资源，但因请求其他资源被阻塞时，对已获得的资源保持不放

  • 不可抢占条件:有些系统资源是不可抢占的，当某个进程已获得这种资源后，系统不能强行收回，只能由进程使用完后自己释放

  • 循环等待条件:若干进程形成环形链，每个都占用对方申请的下一个资源

####java多线程死锁原因

  • 多个线程涉及到多个锁，这些锁存在着交叉，可能会导致一个锁依赖的闭环

  • 默认的锁申请操作是阻塞的

####避免死锁

  • 避免嵌套封锁

  • 只对有请求的进行封锁

  • 避免无限期等待

####竞态条件

  • 产生

    • ﻿当两个线程竞争同一资源时，如果对资源的访问顺序敏感，就称存在竞态条件，导致竞态条件发生的代码区称为临界区

  • 避免

    • 在临界区中使用适当的同步可以避免竞态条件

Java基本数据类型

发表于 2015-01-05   |   分类于 Java   |  

  • boolean:表示范围true/false,大小不确定(java规范中没有明确定义boolean的大小,但在JVM的具体实现中大多数JVM都定义为1B)

  • byte:表示范围-128~127,大小1B

  • char:表示范围0~65535,大小2B

  • short:表示范围-32768~32767,大小2B

  • int:表示范围-2147483648~2147483647,大小4B

  • long:表示范围-2^64~2^64-1,大小8B

  • float:表示范围-3.4E38~3.4E38,大小4B

  • double:表示范围-1.7E308~1.7E308,大小8B

哈希算法相关

发表于 2014-12-24   |   分类于 Algorithm   |  
  • Hash,即散列,就是把任意长度的输入(又叫做预映射,preimage),通过散列算法，变换成固定长度的输出，该输出就是散列值。这种转换是一种压缩映射，也就是散列值的空间通常远小于输入的空间，不同的输入可能会散列成相同的输出，而不可能从散列值来唯一的确定输入值。简单地说就是一种将任意长度的消息压缩到某一固定长度的消息摘要的函数。

  • Hash主要用于信息安全领域中加密算法，它把一些不同长度的信息转化成杂乱的128位编码，这些编码值叫做HASH值，也可以说，hash就是找到一种数据内容和数据存储地址之间的映射关系。

  • 元素牲转变为数组下标的方法：

    • 除法散列法，即
      • index=value%len//len即散列表长度
    • 平方散列法
      • index=(value*value)>>28
    • 斐波那契散列法
      • 将2中的value改为理想乘数
      • index=(value*理想乘数)>>28
      • 理想乘数的计算是通过黄金分割法则得出的，而描述黄金分割法则最经典表达式即斐波那契数列
      • 对于16位整数该理想乘数为40503
      • 对于32位整数该理想乘数为2654435769
      • 对于64位整数该理想乘数为11400714819323198485

  • 散列方法，即碰撞处理，一种是open hashing,也称为拉链法；另一种是closed hashing,也称开地址法,opened addressing。

  • d-left hashing中的d即多个的意思。简化一下，看2-left hashing，指的是将一个哈希表分成长度相等的两部分，分别叫做T1和T2,给T1、T2分别配备一个哈希函数，h1和h2。在存储一个新的key时，同时用两个哈希表函数进行计算，得出两个地址h1[key]和h2[key]。这时需要检查T1中的h1[key]位置和T2中的h2[key]位置，哪一个位置已经存储的key比较多，然后将新key存储到key负载少的位置。如果两边一样多，则把新key存储到左边的t1表中，2-left也由此而来。在查找一个key时，必须进行两次hash，同时查找两个位置。
